# Foundation

## Scope

This document describes the foundation layer of the current RocqSched implementation.

Its scope is intentionally small and currently centers on:

- `theories/Foundation/Base.v`
- `theories/Foundation/Arithmetic.v`

This layer fixes the basic scalar domains, the minimal task/job records, the raw schedule carrier, and a small set of task-model-independent arithmetic lemmas.

It does not define schedule meaning, scheduler interfaces, task-generation semantics, analysis theorems, or operational traces.

## Purpose of the Foundation layer

The foundation layer provides the smallest stable vocabulary shared by the rest of the development.

Its role is to fix:

- the identifier types used for tasks, jobs, CPUs, and time,
- the minimal `Task` and `Job` records consumed by later layers,
- the raw schedule shape used by later semantic developments,
- and small arithmetic facts over `nat` that do not belong to a specific task model.

The design goal is stability. Later layers should be able to grow without forcing churn in these base definitions.

## Core definitions

### Scalar aliases

The current implementation defines four scalar aliases:

- `JobId : Type := nat`
- `TaskId : Type := nat`
- `CPU : Type := nat`
- `Time : Type := nat`

All are currently `nat`. This keeps the base layer constructive and lightweight.

### Task

`Base.v` defines the minimal task-level record:

```coq
Record Task : Type := mkTask {
  task_cost : nat;
  task_period : nat;
  task_relative_deadline : nat;
}.
```

Its fields mean:

- `task_cost`: WCET of each job generated by the task
- `task_period`: period, or more generally the minimum inter-arrival parameter expected by later task-model layers
- `task_relative_deadline`: relative deadline

This record is intentionally minimal. It does not itself enforce properties such as `0 < task_cost`, `0 < task_period`, or deadline constraints. Such validity conditions belong to higher task-model layers.

### Job

`Base.v` also defines the concrete execution instance:

```coq
Record Job : Type := mkJob {
  job_task : TaskId;
  job_index : nat;
  job_release : Time;
  job_cost : nat;
  job_abs_deadline : Time;
}.
```

Its fields mean:

- `job_task`: the generating task identifier; the current convention is `0 = anonymous`
- `job_index`: the `k`-th job of that task, using a `0`-indexed instance number
- `job_release`: absolute release time
- `job_cost`: execution time required by this job
- `job_abs_deadline`: absolute deadline

Like `Task`, this record is deliberately minimal. Positivity and generation-shape constraints are not embedded into the record itself.

### Schedule

The raw schedule carrier is:

```coq
Definition Schedule : Type := Time -> CPU -> option JobId.
```

This is already multicore-shaped, but it is only a raw carrier. The Foundation layer does not include a CPU-count parameter such as `m`, nor does it guarantee boundedness properties like `c < m`. Those are introduced by upper semantic layers such as `valid_schedule`.

## Foundational predicates

The Foundation layer also includes a few close-to-the-data predicates that only talk about jobs and tasks.

### Release-side predicates

```coq
released (jobs : JobId -> Job) (j : JobId) (t : Time) : Prop
pre_release (jobs : JobId -> Job) (j : JobId) (t : Time) : Prop
```

with meanings:

- `released jobs j t := job_release (jobs j) <= t`
- `pre_release jobs j t := t < job_release (jobs j)`

`pre_release` means "not yet released". It is not the same as "waiting in the ready queue"; the name is chosen specifically to avoid that ambiguity.

### Job validity

```coq
valid_jobs (jobs : JobId -> Job) : Prop
```

is defined as:

```coq
forall j, 0 < job_cost (jobs j)
```

So `valid_jobs` is a positive-cost invariant over the job set.

- Zero-cost jobs are not ruled out by the `Job` record itself.
- They are excluded only when a higher result assumes `valid_jobs`.
- This keeps positivity as an explicit predicate rather than a baked-in record invariant.

### Task/job consistency

```coq
valid_job_of_task
  (tasks : TaskId -> Task) (jobs : JobId -> Job) (j : JobId) : Prop
```

expresses only local consistency between a job and the parameters of its generating task:

```coq
let τ := job_task (jobs j) in
job_abs_deadline (jobs j) =
  job_release (jobs j) + task_relative_deadline (tasks τ) /\
job_cost (jobs j) <= task_cost (tasks τ).
```

So this predicate states:

- the absolute deadline is `release + relative deadline`
- the job cost is bounded by the task WCET

Just as importantly, it does not state:

- any periodic or sporadic release pattern
- any spacing property involving `task_period`
- uniqueness of the `job_index`-th job
- monotonicity or non-overlap of generated jobs

Those belong to the task-model layer, not to Foundation.

## Arithmetic lemmas

`theories/Foundation/Arithmetic.v` contains small reusable `nat` lemmas that are independent of any scheduling model.

The current public lemmas are:

- `nat_div2_double`
- `nat_div2_succ_double`
- `nat_div_mul_exact`
- `nat_mod_mul_left`

These are intended to be reused by task-model and analysis layers without pulling scheduling semantics downward into Foundation.

## Public entry points

The default downstream entry points for this layer are:

- `theories/Foundation/Base.v`
- `theories/Foundation/Arithmetic.v`

Clients reading the library from the bottom up should start here to learn:

- the base identifiers,
- the minimal task and job records,
- the raw schedule carrier,
- the basic release and data-consistency predicates,
- and the reusable arithmetic facts that do not belong to a specific task model.

## Design boundaries

This layer includes:

- base identifiers and scalar domains,
- minimal `Task` and `Job` records,
- the raw `Schedule` carrier type,
- simple data-facing predicates attached directly to those records,
- small arithmetic lemmas independent of schedule semantics and task-generation rules.

This layer does not include:

- semantic notions such as `eligible`, `ready`, `completed`, `valid_schedule`, or `feasible`,
- scheduler or algorithm interfaces,
- periodic, sporadic, or jitter-aware job generation,
- busy-window, demand-bound, or interference reasoning,
- operational state machines or trace projection.

Those belong respectively to:

- `design/Semantics.md`
- `design/Abstractions.md`
- `design/TaskModels.md`
- `design/Analysis.md`
- `design/Operational.md`

## Extension points

The current base layer leaves room for later growth without committing to it yet.

The main visible extension directions are:

- richer task records or auxiliary task-model metadata,
- future DAG-aware execution units layered above the current job-centric core,
- alternate finite or abstract CPU domains if the project later needs them.

The current design comment in `Base.v` already anticipates a future split into:

- `Task`: generation rule level
- `Job`: concrete execution instance level
- `Node`: intra-job executable node level for DAG-style models

Under such an extension, the schedule carrier may eventually move from:

- `Time -> CPU -> option JobId`

to something like:

- `Time -> CPU -> option NodeId`
- `Time -> CPU -> option (JobId * NodeId)`

Any such extension should preserve the base role of this layer as a small shared vocabulary rather than push higher-level semantics downward.

## File map

- `theories/Foundation/Base.v`
  Base identifiers, minimal `Task` and `Job` records, the raw `Schedule` type, and simple foundational predicates over jobs and tasks.
- `theories/Foundation/Arithmetic.v`
  Reusable `nat` arithmetic lemmas: `nat_div2_double`, `nat_div2_succ_double`, `nat_div_mul_exact`, and `nat_mod_mul_left`.

## Summary

The Foundation layer is the minimal data layer of the project.

It defines the stable base objects that every higher layer talks about, while deliberately avoiding schedule semantics, scheduler logic, task-generation policy, and operational behavior. Its scope is intentionally narrow: enough to support the rest of the development, but not enough to blur the architectural boundary between raw data, semantics, and analysis.
